Project: bevy_screeps_lua
Language: Rust
Engine: Bevy
Scripting: Lua
Target: Desktop first
Map size: 256 x 256 world tiles
Room grid: 32 x 32 rooms
Tiles per room: 8 x 8 world tiles
Default players: 32 AI factions
Default mode: GOD mode enabled
Primary design goal: a Screeps-like autonomous simulation where units are controlled by Lua scripts loaded from per-AI directories, not by direct user-issued unit commands.

==================================================
1. HIGH LEVEL GAME SPEC
==================================================

1.1 Core concept

This game is a persistent RTS/simulation sandbox inspired by Screeps.

The player is not issuing direct move/attack/build orders to each unit.
Instead, each faction runs Lua scripts that decide how its creeps behave.

Each AI faction has:
- its own Lua files
- its own save/state namespace
- its own units
- its own buildings
- its own resource stockpile
- its own diplomacy/visibility state

The simulation runs continuously in ticks.
Lua logic is executed on scheduled hooks each tick.

The default game world contains 32 AI factions, each with its own folder:
- ai/ai_00/
- ai/ai_01/
- ...
- ai/ai_31/

Each folder contains Lua scripts such as:
- main.lua
- roles/harvester.lua
- roles/builder.lua
- roles/fighter.lua
- config.lua

The human user starts in GOD mode by default:
- can inspect entire map
- can inspect any unit
- can inspect any room
- can inspect any AI script status
- can pause/resume/speed up simulation
- can spawn units/resources/buildings for testing
- can open debug windows and logs
- can reload Lua scripts live

The user is mainly a simulator operator, observer, debugger, or sandbox god, not a traditional RTS commander.

1.2 World structure

World dimensions:
- 256 x 256 world tiles total

Room structure:
- 32 x 32 rooms total
- each room is 8 x 8 world tiles

Therefore:
- world width = 32 rooms * 8 tiles = 256
- world height = 32 rooms * 8 tiles = 256

Each room stores:
- terrain layout
- mine nodes
- building list
- unit list
- dropped resources
- visibility state
- room owner/controller state
- room memory/debug metadata

1.3 Mine system

Each room contains 2 or 3 mine nodes.

There are 10 mine types total:

1. Power Mine
   - the main life resource for creeps
   - required for survival and upkeep
   - used like energy/food/fuel hybrid
   - consumed by creeps periodically

2. Iron Mine
   - used for basic metal structures
   - used for armor, frames, melee parts

3. Copper Mine
   - used for wiring, circuits, simple electronics

4. Silicon Mine
   - used for compute structures and logic cores

5. Crystal Mine
   - used for sensors, optics, advanced buildings

6. Carbon Mine
   - used for lightweight components and fuel processing

7. Stone Mine
   - used for walls, foundations, simple defensive structures

8. Sulfur Mine
   - used for explosives, ammo, chemical warfare buildings

9. Water Mine
   - used for cooling, life support, bio processing

10. Biomass Mine
   - used for growth vats, food synthesis, creep organics

Rules:
- every room has 2 or 3 mine nodes
- at least one nearby Power Mine should exist within reasonable expansion distance, but not every room must contain one
- mine type distribution should encourage territorial expansion
- some rooms may be strategically valuable because of rare combinations
- mines regenerate over time
- mines may have max extraction rate
- mines may be exhausted temporarily and recover later

1.4 Units

Units are called screeps in this spec, but internal code may use the generic term creep.

A screep is composed from components/parts.
The available part families are:

- Move
- Work
- Fight
- Mine
- Build
- Eat
- Transport

Meaning:
- Move: movement capability and speed
- Work: general task execution capability
- Fight: combat capability
- Mine: extraction efficiency
- Build: construction efficiency
- Eat: metabolism/life support capacity
- Transport: carry capacity

Each screep must consume Power resource to stay alive.
Without sufficient Power:
- screep becomes weak
- movement slows
- task performance drops
- eventually the screep dies

Possible derived stats:
- hp
- speed
- cargo capacity
- upkeep cost
- mining rate
- build rate
- melee damage
- ranged damage if later added
- repair rate
- sensor range if later added

Screeps are autonomous.
They do not execute direct user commands.
Instead:
- user may inspect
- user may override in GOD mode for debugging
- AI Lua decides normal behavior

1.6 AI / Lua model

Every faction has its own Lua VM context or sandboxed environment.
It loads scripts from its own directory.

Example:
- ai/ai_00/main.lua
- ai/ai_00/roles/harvester.lua
- ai/ai_00/roles/builder.lua

Lua is responsible for:
- deciding unit roles
- deciding target mines
- deciding when to build
- deciding when to fight
- deciding logistics priorities
- deciding where to expand
- reacting to world observations

Lua is not allowed to directly mutate raw ECS state.
Instead it uses Rust-exposed API handles and commands, for example:
- get_owned_units()
- get_visible_rooms()
- issue_move(unit_id, x, y)
- issue_mine(unit_id, mine_id)
- issue_build(unit_id, blueprint, x, y)
- get_stockpile(resource_type)

The command model should be:
- Lua reads a safe snapshot or query interface
- Lua produces intents/commands
- Rust validates and applies them during simulation steps

1.7 Tick model

Simulation proceeds in discrete ticks.

Example phases per tick:
1. input/debug actions
2. script scheduling
3. Lua AI execution
4. command validation
5. movement
6. mining
7. transport
8. building
9. combat
10. upkeep and power consumption
11. regeneration and economy
12. event log flush
13. UI refresh

Each AI may run every tick or on a staggered schedule for performance.
Possible default:
- every AI gets a budget
- heavy logic may be split across ticks
- Lua instruction count limit to prevent one AI from freezing game

6. LUA SCRIPT SPEC
==================================================

6.1 Standard AI folder layout

ai/ai_00/
├─ main.lua
├─ config.lua
├─ roles/
│  ├─ harvester.lua
│  ├─ builder.lua
│  ├─ fighter.lua
│  ├─ hauler.lua
│  └─ scout.lua
└─ libs/
   └─ util.lua

6.2 main.lua expected interface

The Rust side should look for these optional functions:
- on_init()
- on_tick()
- on_unit_spawned(unit_id)
- on_unit_died(unit_id)
- on_room_visible(room_id)

Minimal required:
- on_tick()

6.3 Example Lua behavior model

on_tick():
- read stockpile
- count units by role
- if power income too low, spawn more harvesters
- send miners to power mines
- send haulers to depots
- if storage high enough, spawn builders
- if enemy nearby, spawn fighters

6.4 Lua limitations

- no direct filesystem access except its own allowed files if enabled
- no OS command execution
- no raw network access
- instruction budget per tick
- safe Rust API only
- errors isolated per faction

pub type FactionId = u16;
pub type CreepId = u32;

    Copper,
    Silicon,
    Crystal,
    Carbon,
    Stone,
    Sulfur,
    Water,

}

    MoveTo { creep_id: CreepId, x: i32, y: i32 },

    Attack { creep_id: CreepId, target_id: u32 },
    Transfer { creep_id: CreepId, target_id: u32, resource: ResourceType, amount: u32 },

}

}

13. NOTES FOR FILE SIZE CONTROL
==================================================

To avoid huge files:
- keep one subsystem concern per file
- split actions by domain: movement, mining, combat, building
- split Lua API by topic
- split UI panels into separate files
- keep worldgen split into terrain, mines, and faction spawn logic
- keep building logic split by structure family
- keep debug tools isolated from core sim
- keep save/load isolated from runtime systems

Do not place all creep logic into one giant creeps.rs.
Do not place all Lua bindings into one giant lua.rs.
Do not place all UI into one giant ui.rs.
